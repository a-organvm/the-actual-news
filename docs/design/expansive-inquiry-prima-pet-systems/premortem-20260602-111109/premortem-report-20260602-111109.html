Prima Pet System Evolution Failed Six Months From Now

Prospective hindsight on evolving Prima from a visual Codex pet into a trustworthy ambient state familiar.

Most Likely Failure

Role ambiguity: the pet remains decoration, state surface, companion, and possible control layer all at once.

Most Dangerous Failure

Capability creep before governance: pets become executable context surfaces before signing, sandboxing, and permissions exist.

Hidden Assumption

Emotional vividness will naturally mature into system trust. It will not.

Revised Plan

  1. Define the role hierarchy first: visual pet, ambient state surface, user-configured state mapping, and deferred capability-bearing companion.
  2. Create a manifest-first package standard with version, compatibility, state inventory, preview assets, checksums, provenance, and license.
  3. Bind pet state only to authoritative runtime state, including explicit unknown and stale states.
  4. Validate small-scale readability, reduced motion, contrast, theme safety, transparency, and fallback rendering before runtime expansion.
  5. Separate the general pet standard from Prima's phasing, density, swarm, and negative-space vocabulary.
  6. Freeze scripts, hooks, context reads, network behavior, and memory until a formal security model exists.

Pre-Launch Checklist

  • A reviewer can identify each supported workflow state from Prima at intended UI scale, including reduced-motion mode.
  • pet.json declares supported states, assets, version, compatibility, provenance, checksums, license, and no capability fields.
  • Runtime state mapping is driven by one authoritative enum with stale and unknown handling.
  • A second non-Prima pet can express the same workflow states without using Prima-specific vocabulary.
  • Governance review confirms no package field carries private user context, code execution, network behavior, or unapproved hooks.
