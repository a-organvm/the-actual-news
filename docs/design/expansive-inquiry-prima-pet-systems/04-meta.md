## System Diagnosis

Scope sharpened the inquiry from "make a pet" into "define a companion and its system boundary." Logic exposed the product architecture: asset package, runtime, state protocol, distribution, and safety. Mythos preserved the reason Prima should not become a generic status icon: he is a familiar of transformation, not a notification badge. Bridge found implementation analogies that make the design more precise, especially choreography and stage lighting.

The main tension is productive: Prima wants to be alive, but the system must not imply agency it cannot honor. That means design should make process visible without making false promises about autonomy.

## Feedback Loop

1. Visual audit loop
   - Inspect current sprites for legible negative space, lean silhouette, phasing, and state readability.
   - Prune frames where the swarm reads as smoke, goo, or generic magical blur.
   - Regenerate only weak states rather than restarting the whole asset.

2. Runtime semantics loop
   - Define a minimal pet state vocabulary: idle, active, waiting, blocked, done, error.
   - Map each state to posture, particle density, tempo, and optional color accent.
   - Test whether the state is readable with animation muted and with color inaccessible.

3. Packaging loop
   - Treat `pet.json` as a manifest, not just metadata.
   - Add version, author, license, sprite dimensions, supported states, and optional provenance fields when the client supports them.
   - Validate package shape before distribution.

4. Governance loop
   - Declare what pet behavior is allowed to observe.
   - Separate visual personalization from runtime context access.
   - Require explicit consent for any behavior beyond local animation state.

## Three Adaptations

1. Prima should be designed as a material system first and a character second.
2. Pet packages should evolve toward capability declarations before they evolve toward scripts.
3. The pet's most important future API is not "say things" but "express state honestly."
